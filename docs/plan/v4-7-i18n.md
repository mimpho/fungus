# Plan v4.7 — i18n / Traducciones

## Contexto

La infraestructura i18n ya existe en el frontend:
- `AppContext.jsx` gestiona `lang` (es/ca/en) y expone `t` con las strings traducidas
- `src/data/i18n.js` contiene las traducciones ES/CA/EN
- El selector de idioma funciona en la página de Perfil

El trabajo de v4.7 es **completar y sanear lo que ya hay**, no montar una capa nueva.

---

## Alcance

### 1. Frontend — auditoría de strings hardcoded

Recorrer todos los componentes y páginas buscando texto en español que no pase por `t()`.
Archivos a revisar: `pages/`, `components/`, `components/modals/`, `articles/`.

Para cada string encontrado:
- Añadir la key en `i18n.js` con sus tres valores (ES / CA / EN)
- Reemplazar el literal en el componente por `t('KEY')`

### 2. Frontend — completar `i18n.js`

Verificar que las keys ya existentes tienen los tres idiomas cubiertos.
CA y EN son los más probables de tener huecos — completar los valores que falten.

### 3. Contenido en BD — criterio para v4.7

Los textos almacenados en BD (descripciones de zonas, `diff` de confusiones, sinónimos, descripción de especies) están en español y son contenido especializado, no strings de UI.

**Estrategia**: el texto en español es el canónico. Para CA/EN se usa el patrón `extra_data` ya existente en el schema — sin cambio de schema, sin tabla nueva.

Ejemplo para confusiones:
```json
extra_data: {
  "confusions": [...],
  "confusions_ca": [...],
  "confusions_en": [...]
}
```

**Para v4.7 MVP**: el contenido BD queda en ES. Las traducciones CA/EN de contenido se añaden solo si hay recursos para redactarlas — no es un bloqueante para la epic.

### 4. Backend — endpoint de traducciones (opcional)

Si el frontend necesita strings dinámicos desde BD (etiquetas de categorías, nombres de tipos de bosque, etc.), el patrón es un endpoint ligero que sirva el objeto `i18n.js` por locale, o simplemente se mantiene todo estático en el frontend.
No se crea una tabla `translations` en BD.

---

## Lo que NO entra en v4.7

- Auth, login, favoritos → v5
- Apps móviles → v6
- Tabla `translations` en BD → descartado (overengineering para este volumen)
- Traducción automática del contenido BD → fuera de scope MVP

---

## Entregable mínimo

- `i18n.js` completo: sin keys sin traducir, sin strings hardcoded en componentes principales
- La app funciona en ES / CA / EN sin textos rotos ni fallbacks visibles
- Decisión documentada sobre contenido BD (canónico ES + extra_data opcional para CA/EN)
