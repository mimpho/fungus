# Plan 4.7 MVP — I18n / Translations (DB-backed proposal)

Context
- Current codebase lacks a centralized i18n layer; UI strings appear hardcoded.
- Goal: enable translations for UI with a simple, scalable approach for future MVPs.

Objectives
- Introduce a minimal i18n layer with a translations store in DB (locale, key, value).
- Wire a lightweight i18n fetcher in frontend to query translations by locale and key.
- Start with ES and EN translations for core UI texts (labels, buttons, messages).
- Keep MVP scope small to avoid large refactors; ensure existing UI remains intact.

Data model (DB)
- translations
  - id: PK
  - locale: string (e.g., 'es', 'en')
  - key: string (e.g., 'LOGIN', 'LOGOUT', 'FAVORITES')
  - value: string
  - created_at, updated_at

Migration plan (high level)
- 005_i18n_translations.md: create translations table
- Seeds: 005_a11y_es_en.md: initial translations for common keys

Frontend integration (MVP)
- Create a small i18n hook or context to fetch translations by locale
- Replace a few static labels with translation lookups (e.g., in login UI)
- Use locale selection (store or user preference)

Security & Access
- Ensure translations are served from API under a locale header or query param
- Cache translations in memory on frontend to reduce repeated calls

Testing
- Basic checks: en/es translations available for core keys
- Ensure non-translated keys fallback to English

Docs & Deliverables
- docs/plan/v4-7-i18n.md (this doc)
- backend/migrations/005_i18n_translations.md (template)

Timeline (rough)
- Week 1: DB schema & migration scaffolding; frontend i18n hook
- Week 2: UI translations for core flows (login, favorites)
- Week 3: QA, polish, docs

Note: Google login remains for 4.7.1.
